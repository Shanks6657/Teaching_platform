# 《在线教学与实训平台》软件开发计划书

## 1. 项目简介
**项目名称**：在线教学与实训平台
**项目目标**：开发一个一体化的在线教学与实验平台，满足高校师生的在线教学与实训需求。实现课程管理、在线实验、自动评测、作业布置与批改、成绩统计等核心功能，打造真实教学场景，确保评测结果的准确性与系统的稳定性。


## 2. 系统用户与核心角色
- **教师端**：创建和管理课程、布置作业/实验、在线批改、成绩统计、题库/课件管理。
- **学生端**：选择课程、在线学习、完成代码实验/作业并提交自动评测、查看成绩和解答。
- **系统管理员端**：用户管理（师生账号批量导入与权限分配）、系统配置、日志查看。

## 3. 核心功能模块划分
### 3.1 基础模块
- **用户认证与授权**：暗色系风格的登录/注册功能，支持教师与学生的身份区分，提供鉴权机制。
- **个人中心**：资料修改、密码重置、系统使用倾向设置。

### 3.2 教师端功能模块
- **课程管理**：创建新课程（支持设置封面、说明、大纲等），管理本人的课程列表。
- **教学计划安排**：设置课程学习路径。
- **实验/作业发布**：发布带有自动评测机制（含测试用例配置）的代码实验或常规作业，配置截止时间。
- **批改与成绩统计**：查看学生提交记录，人工或自动批改作业，支持成绩单批量导出。

### 3.3 学生端功能模块
- **首页**：展示自己的选课列表、近期DDL（截止日期）的作业及通知。
- **课程学习**：浏览教师上传的课件、资源。
- **在线实验区**：集成WebIDE或轻量级在线代码编辑器，支持在线编写代码并提交运行测例。
- **成绩与反馈反馈**：查看测例执行结果（AC/WA/CE/RE等），获取分数和教师评价。

### 3.4 评测服（Judge Server）模块
- **沙盒运行环境**：隔离运行学生提交的代码，支持多语言（优先Java、C/C++、Python）。
- **资源限制**：限制时间、空间（CPU/内存使用）。
- **评测引擎**：将学生输出与标准答案对比，返回状态信息和错误日志。

## 4. 技术选型
- **后端框架**：Java 8, Maven, Spring Boot , MyBatis
- **前端技术**：HTML/CSS/JS 结合模板引擎（Thymeleaf），暗色风格UI（Cyberpunk CSS）
- **数据库**：MySQL (核心业务数据), Redis (缓存及Session)
- **评测引擎**：Docker 或基于开源评测机改造

## 5. 项目架构与规范
- 基础包名：`org.example`
- 入口类：`org.example.TeachingPlatformApplication`
- 开发规范：严格遵循Java 8语法和开发实践，使用UTF-8编码，依赖统一通过`pom.xml`管理。
- 目录结构：标准Maven目录结构管理，资源文件按约定放置于`resources/static`与`resources/templates`中。

## 6. 开发生命周期与里程碑计划
### 阶段一：需求调研与规划 (第7周)
- 需求获取分析(Elicitation)：确认基本需求与用户故事。
- 需求精化与规约(Refinement & Specification)：确定所有交互细节与字段,确定前端整体风格。
- 需求确认与验证(Validation & Verfication)：确认需求文档，确保无歧义。
- 梳理数据库ER图与核心表结构。

### 阶段二：系统架构搭建与基础模块 (第8周)
- 初始化Spring Boot项目环境及Maven配置。
- 数据库建表与ORM框架集成。
- 完成用户注册、登录、权限校验（鉴权）模块开发。
- 完成学生首页、教师首页基础数据展示。

### 阶段三：核心业务模块开发 (第9-11周)
- **教师端开发**：课程的增删改查、实验和题库的管理配置。
- **学生端开发**：课程加入、浏览作业列表、在线编码与提交视图的开发。

### 阶段四：评测系统集成 (第11-12周)
- 调研开源Judge Sandbox并实现评测微服务端。
- 主系统与评测端的RPC或消息队列对接。
- 编写测试用例保证评测准确性和系统并发处理能力。

### 阶段五：系统测试与优化 (第13周)
- 全流程集成测试（测试用例全覆盖）。
- 安全测试（防代码注入、资源耗尽型攻击等）。
- UI/UX走查：验证风格的美观度与交互体验。

### 阶段六：上线部署与交付 (第14周)
- 编写部署文档（数据库初始化、项目打包构建）。
- Docker容器化部署与线上稳定性监控调试。
- 向客户交付并进行使用培训。

## 7. 风险评估与对策
- **评测机安全风险**：学生可能提交恶意代码。对策：严格基于Docker沙盒进行隔离，配置最小执行权限和硬件资源配额。
- **高并发风险（如收作业时间点）**：大量学生集中提交造成的拥堵。对策：采用Redis缓存热点数据，评测提交采用异步队列削峰填谷。
- **易用性风险**：在线IDE的体验欠佳。对策：集成成熟前端代码编辑器组件（如Monaco Editor）。
